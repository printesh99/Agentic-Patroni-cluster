# PostgreSQL Enterprise Console — Architecture & Project Documentation

**Version:** July 2026 · violet enterprise theme
**Stack:** React 18 (pre-built, no bundler at runtime) · Apache ECharts · plain CSS · static file hosting

---

## 1. What This Project Is

The PostgreSQL Enterprise Console is a single-page web application for monitoring and operating PostgreSQL clusters running on OpenShift/Kubernetes under the Crunchy Data PGO operator with Patroni high availability, PgBouncer connection pooling, and pgBackRest backups to S3.

It gives four audiences one console:

| Audience | Screens they live in |
|---|---|
| **Executives** | Executive Overview, SLA, Cost Showback, Management Scorecard |
| **DBAs** | SQL Insight, Memory/SGA, Performance views, Vacuum/Bloat/Index advisors |
| **Platform / SRE** | Cluster/Patroni, Replication, WAL & Archive, Backups, DR Readiness, Storage/Platform health |
| **Security & Compliance** | Security Posture, Compliance Reports, Access Recertification, pgaudit |

## 2. Runtime Architecture

```
Browser
└── index.html  (or index.preview.html for standalone demo)
    ├── styles.css                  ← all styling, incl. theme + polish layers
    ├── vendor/echarts.min.js       ← chart engine
    ├── dist/chart3d.js             ← global chart enhancer (wraps echarts.init)
    ├── dist/countup.js             ← KPI number animation
    ├── dist/data.js                ← API helpers (clusterPath, fetch wrappers)
    ├── dist/components.js          ← shared UI kit + navigation model
    ├── dist/topology_full.js       ← ClusterArchitecture diagram component
    ├── dist/memory_sga.js          ← Memory / SGA module
    ├── dist/sql_insight.js         ← SQL Insight module
    ├── dist/appmon_charts.js       ← App Monitoring chart components
    ├── dist/overview_charts.js     ← Overview trend fallbacks
    ├── dist/module_charts.js       ← shared viz kit (window.HbzViz) + 6 modules
    ├── dist/module_charts2.js      ← perf/cluster/replication charts
    ├── dist/module_charts3.js      ← capacity/cost/security/ops charts
    ├── dist/module_charts4.js      ← app activity + metrics compare
    ├── dist/ai_ops.js              ← AI Operations band (5 screens)
    ├── dist/health_grid.js         ← Cluster Health Grid (NOC/exec + kiosk)
    ├── dist/sql_insight2.js        ← SQL Insight tabs (DB Load/Queries/Compare/Blocking)
    ├── dist/drilldowns.js          ← interactive drill-downs (cluster, memory, backups, alerts, pgbouncer)
    ├── dist/<screen>.js …          ← one file per screen family
    └── dist/app.js                 ← router, layout shell, screen mounting
```

Key properties:

- **No build step at runtime.** Every `dist/*.js` file is a classic script (IIFE or top-level functions). Load order in `index.html` is the dependency graph.
- **`window` is the module system.** Screens and chart components register themselves as globals (`window.MemorySgaScreen`, `window.HbzViz`, …); `app.js` mounts them by route key.
- **Cache busting** via `?v=<timestamp>` query params on every script/style tag — bump them on each deployment.

## 3. Data Flow — Live-First With Badged Fallback

Every data-driven element follows one universal pattern:

```
fetch live endpoint ──succeeds──▶ render live data (no badge)
        │
        └──fails / empty──▶ render representative sample data
                             + amber pill: "representative sample"
```

- API base: `/api/v1/clusters/{clusterId}/…` (helper: `clusterPath(path)` in `dist/data.js`)
- Sample data is generated by seeded PRNGs (`seeded(n)`) so it is **stable across reloads** — no flickering demo numbers
- The badge is the single source of truth for "this is not live" — never remove the badge logic when wiring real endpoints

### Endpoints consumed by the new modules

| Module | Endpoint / source | Payload shape |
|---|---|---|
| Memory / SGA | `GET …/config/parameters` | `{parameters: [{name, setting, unit}]}` |
| SQL Insight (ASH) | activity sampler (to be wired) | per-minute `{ts, wait_class, count}` |
| SQL Insight (top SQL) | pg_stat_statements (to be wired) | calls, mean/total time, rows, query text |
| AppMon trend | ui metrics API | `{series: [{label, points: [[ts, val]…]}]}` |
| AppMon domain | domain API | `{dead_tuples: […], sessions: […]}` |
| AppMon repl | repl API | `{slots: [{slot, retained_wal_bytes}…]}` |

## 4. The Architecture Diagrams (in-app)

### 4.1 Cluster Topology (`dist/topology_full.js` → Cluster/Patroni screen)

Shows how the whole cluster hangs together, left-to-right by traffic direction and top-to-bottom by plane:

```
Applications ──SQL·5432──▶ PgBouncer ──pooled──▶ K8s Service
                                                     │ read-write
                                                     ▼
        ┌───────── PostgresCluster · OpenShift namespace ─────────┐
        │  Leader (primary) ──streaming replication──▶ Standbys   │
        │        │                                        │       │
        │        └────────── Patroni HA bar ──────────────┘       │
        │        (leader election · health checks · failover ·    │
        │         DCS: Kubernetes endpoints)                      │
        └──────────────────────────────────────────────────────────┘
             │ WAL archive · backups
             ▼
   PGO Operator ··reconciles CR··▶ [band]
   pgBackRest repo-host ──pgBackRest──▶ S3 bucket
```

**Implementation:** nodes are absolutely-positioned `<div>` cards (pixel space, ResizeObserver re-measures on resize); edges are SVG cubic Bézier paths with per-color arrowhead markers and an animated `stroke-dashoffset` flow. Edge colors are semantic and mirrored in the legend:

- violet `#7c3aed` — client traffic
- green `#26a65b` — streaming replication
- amber `#b7791f` — backups / WAL
- violet dashed — operator control
- gray dashed — Patroni coordination

With live Patroni data, pods show real names, states, timelines and lag; without it, ghost placeholders render (dashed borders) so the diagram always teaches the architecture.

### 4.2 Memory Topology (`dist/memory_sga.js` → Memory / SGA screen)

Oracle-analogue view of PostgreSQL memory, three bands top-to-bottom:

1. **Shared Memory (≈ SGA)** — one per instance, allocated at startup: shared_buffers pool, WAL buffers, CLOG/SLRU, lock space — each with a fill gauge
2. **Per-Backend Memory (≈ PGA)** — multiplies per connection: work_mem (per sort!), temp_buffers, maintenance_work_mem
3. **OS & Disk** — effective_cache_size (planner estimate, *not* an allocation), data files, pg_wal → archive → S3

Flow labels between bands explain the page lifecycle (backends → buffer pool → bgwriter/checkpointer → disk). Values come from live `pg_settings` where available.

### The `wrap()` extension pattern

New chart rows were added to existing screens **without editing the screen source**, via a higher-order wrapper in `dist/module_charts.js`:

```js
function wrap(name, buildCharts) {
  var orig = window[name];
  window[name] = function (props) {
    var charts = buildCharts(props);       // may return null → passthrough
    if (!charts) return e(orig, props);
    return e(React.Fragment, null,
      e("div", {className: "page"}, charts),
      e(orig, props));
  };
}
```

- Wrappers run at script load; `module_charts*.js`, `sql_insight2.js` and `drilldowns.js` must load **after** all screen files and **before** `app.js`
- Wraps compose: a screen can be wrapped several times (e.g. `ClusterScreen` — charts in `module_charts2.js`, member drill-down in `drilldowns.js`); returning `null` defers to the inner wrap
- `drilldowns.js` adds interactive layers: Cluster member drill-down + switchover dry-run, Memory/SGA component inspector, Backups restore-point timeline + guided restore test, Alert correlation (±30 min metric window + related events), PgBouncer pool slicer + queue-wait histogram
- When porting to a real build system, replace wraps with direct chart integration inside each screen component

## 7. Chart System

### chart3d.js — the global enhancer
Wraps `echarts.init` so **every** chart in the app, old or new, gets:
- Bars: vertical gradient fill (lighten→darken of the base color), rounded top caps, drop shadow → the "3D extruded" look
- Lines: soft glow shadow; areaStyle opacity converted to a vertical fade gradient
- Pies: lifted slices (shadow), white rims, rounded corners
- Tooltips: dark-violet theme (`rgba(30,23,64,.95)`, radius 8, no border, 11px `#eae7f3` text) applied wherever a chart doesn't override it

### Shared viz kit — `window.HbzViz` (module_charts.js)
`EBox` (ref-based ECharts mount, `animation:false`), `card(title, meta, body)`, `donut`, `gauge`, `baseAxis(x)`, `hours/days/months`, `seeded`, `hump` (business-hours curve generator), `badge`, `wrap`.

### Conventions every chart follows
- axis labels 10px; legend bottom (12×8 swatches); grid ~`{top:24, right:14, bottom:40, left:46}`
- threshold marklines: dashed pink `#ef476f` with 10px inside-end-top label
- risk coloring: ≥ high → pink, ≥ mid → amber, else violet
- horizontal bars: barWidth 13, value labels at right, mono y-labels
- legends wrapping to 2 rows in half-width cards need `grid.bottom ≥ 62`
- log scales for calls/latency distributions (SQL scatter + histogram)

## 9. Deployment Notes

1. Serve the `static/` folder from any web server; `index.html` expects the API at the same origin (`/api/v1/…`). `index.preview.html` is the self-contained demo variant.
2. Bump every `?v=` timestamp on deploy (or replace with your build hash).
3. Script order matters: vendor → chart3d → countup → data/components → feature modules → module_charts 1-4 → app.js.
4. The app degrades gracefully with zero live endpoints — every screen renders with badged samples, which is also the recommended local-dev experience.
